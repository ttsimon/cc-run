# ccr chain 设计（v0.3）

> 状态：设计已定稿（2026-06-09 brainstorm 通过）。**计划归属 v0.3**——v0.2 先发「快点小功能」批（见 `2026-06-09-ccr-quickwins-design.md`），chain 作为下一个大版本。
> 前置依赖：A 档小功能（`doctor`/`test` 后端体检、per-profile 默认参数）随本版一起做。

## 定位

多后端 agent 流水线：每段挂不同 provider，把官方 subagent / Agent SDK 不好做的「异构后端串联」做成生态位。

填的真空白：**用户想把不同模型串起来（各段用各自最擅长的后端），但目前没有趁手工具，只能从头到尾用一个模型。** chain 不是重造官方已有的 subagent，而是补「跨后端流水线」这个缺口。

## 首条参考链（也是内置模板）

```
规划(强模型) → 实现(便宜模型) → 审查(另一家)
```

设计哲学：**智力前置到规划**。强模型把方案/步骤拆到足够细，实现段就退化成「照着干」的体力活，便宜后端也能扛；最后用一个审查模型兜底。这与 superpowers `writing-plans → executing-plans`（先写详尽计划、再机械执行）同源。

## 引擎（三件事）

### 1. 分段执行
- 每段 = 一次无头 `claude -p`，跑完即退 = 天然交棒边界。
- env 注入复用现有 ccr 的 provider 机制（每段在 yaml 里指定用哪个 profile/后端）。

### 2. 交棒
- **共享同一个 git 工作目录**，文件天然跨段留存。
- 上段把**结构化产出写文件**：
  - 规划段 → `docs/plans/<task>.md`
  - 审查段 → `findings` 文件 + 判定字段（`pass` / `needs-work`）
- 下段 prompt 用 `{{prev.output}}` 注入上段 stdout 的**一句指路**（如「计划已写到 `docs/plans/xxx.md`，去读它照着干」），**只指路、不搬运内容**。
- **规划段强制结构化输出**：阶段 / 任务 / 验证三件套，钉进规划段的 prompt 模板，保证便宜模型在实现段跑不偏。

### 3. 人在环里
- 默认 **段间放行**：每段停在 `⏸`，可选「放行 / 改指令 / 跳过 / 退出」。
- `⏸` 时用户可直接编辑工作目录里的计划 / findings 文件，再放行。这是放行真正有用的地方。
- `--auto` 关停顿、一条道跑到黑。

## 审查收场

- 审查段固定动作：写 `findings` + 判定，停在 `⏸`。
- 收场靠「放行」这一个旋钮，**不是引擎里的循环**：
  - 判定 `pass` → 放行即收链，结束。
  - 判定 `needs-work` → 用户决定：放行进「修复段」（读 findings 照着改），或直接停、手动接。
- 链 yaml 里**可选**地定义一个修复段——它是「又一个普通段」，复用同一套放行引擎；不会自动重跑、不会循环。想再审一轮就重跑链，那也是手动。
- `--auto` 模式：定义了的段一条道跑到黑，findings 与判定照写但**不据判定分叉**（据判定跳过/重跑 = YAGNI 掉的条件分支）。判定在 auto 模式纯留痕。

## 执行可观测性

- **可观测性**：段以 stream-json 跑，ccr 实时渲染工具调用与结果，详细度 `-q`/默认/`-v` 可调；放行点附 `diff --stat` 与耗时；非 TTY 自动降级。共享样式层 `internal/ui`。详见 `specs/2026-06-11-ccr-chain-observability-design.md`。

## 安全：四层纵深

1. 可插拔隔离（Isolator）：git 目录用临时 worktree（每段 ccr 兜底提交，避免 agent 不提交导致丢失）；非 git 目录用 copydir 快照。结束三态——跑完且审查 pass → 本地 merge 合回当前分支并删临时分支；needs-work / 报错 / 用户退出 → 保留成果并打印取回路径，**绝不静默销毁**。详见 `specs/2026-06-11-ccr-chain-isolation-design.md`。
2. 每段 `allow-tools` 白名单。
3. `PreToolUse` 钩子做命令红线黑名单（ccr 内置默认 + 用户在 yaml 追加），命中即 halt。
4. 写操作圈在工作目录内。

> ⚠️ Windows 真沙箱弱，主要靠 ①（隔离 + 成果交回）+ ③（钩子黑名单）兜底。

## yaml 谁写（v0.3 范围）

- 引擎只认 yaml。
- 做：手写 + `ccr chain init` 出模板 + 1~2 个内置模板（规划→实现→审查）。
- **推迟**：`ccr chain new` 向导、`ccr chain plan "目标"`（让规划 agent 自动生成链）。

## 怎么算成功（验收锚点）

跑通这条链一次，产出**不差于一个强模型从头干到尾**，但**实现段用便宜后端**——质量靠规划的结构化 + 审查兜底守住，成本/灵活性是净赚。这一条是判断「方向到底成不成」的实测口径。

## 明确不做（YAGNI）

- DAG / 分支 / 循环 / 重试（不造 LangGraph）。
- 交互式接力（段内对话）。
- AI 自动生成链。
